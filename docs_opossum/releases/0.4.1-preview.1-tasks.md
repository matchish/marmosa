### Why a minor-version bump?

In pre-1.0 SemVer, a breaking public API change warrants a minor bump (0.4.x → 0.4.1).
The change is small and intentional — it unlocks a third operational mode that a boolean
can never express — but it breaks all existing config files and any code that sets the
property directly.

## API Design

### New enum (new file: `src/Opossum/Projections/AutoRebuildMode.cs`)

```csharp
/// <summary>
/// Controls how the projection daemon handles missing or stale projections at startup.
/// </summary>
public enum AutoRebuildMode
{
    /// <summary>
    /// The daemon starts but never triggers a rebuild on startup.
    /// Use in production when projections are managed manually or via admin endpoints.
    /// Was: EnableAutoRebuild = false
    /// </summary>
    None,

    /// <summary>
    /// Only projections whose checkpoint file is absent are rebuilt.
    /// Projections that were already rebuilt (even against an empty store) are skipped.
    /// This is the recommended default for most deployments.
    /// Was: EnableAutoRebuild = true (the only mode previously available)
    /// </summary>
    MissingCheckpointsOnly,

    /// <summary>
    /// All projections are rebuilt from scratch on every startup, regardless of checkpoints.
    /// Use in development when iterating on projection logic and wanting a clean slate
    /// on each restart, or after deploying a projection fix that requires a full replay.
    /// Was: not expressible with a boolean.
    /// </summary>
    ForceFullRebuild
}
```

### Updated property on `ProjectionOptions`

```csharp
// Before
public bool EnableAutoRebuild { get; set; } = true;

// After
public AutoRebuildMode AutoRebuild { get; set; } = AutoRebuildMode.MissingCheckpointsOnly;
```

Note the **property rename** (`EnableAutoRebuild` → `AutoRebuild`). With an enum value
the old name reads awkwardly (`EnableAutoRebuild = None`). The rename is part of this
breaking change and costs nothing extra.

### Migration guide for existing users

| Old config / code | New equivalent |
|---|---|
| `"EnableAutoRebuild": true` | `"AutoRebuild": "MissingCheckpointsOnly"` |
| `"EnableAutoRebuild": false` | `"AutoRebuild": "None"` |
| `options.EnableAutoRebuild = true;` | `options.AutoRebuild = AutoRebuildMode.MissingCheckpointsOnly;` |
| `options.EnableAutoRebuild = false;` | `options.AutoRebuild = AutoRebuildMode.None;` |

Default behaviour is **unchanged** — `MissingCheckpointsOnly` is the new default,
which is exactly what `true` did before.

## Task Detail

### Task 1 — New `AutoRebuildMode.cs`

Create `src/Opossum/Projections/AutoRebuildMode.cs` with the enum shown in the **API
Design** section above. No `using` statements needed (no external dependencies).

### Task 2 — `ProjectionOptions.cs`

Replace:
```csharp
/// <summary>
/// Enable automatic projection rebuilding on startup if checkpoint is missing
/// Default: true
/// </summary>
public bool EnableAutoRebuild { get; set; } = true;
```

With:
```csharp
/// <summary>
/// Controls how the projection daemon handles projections at startup.
/// Default: <see cref="AutoRebuildMode.MissingCheckpointsOnly"/>
/// </summary>
public AutoRebuildMode AutoRebuild { get; set; } = AutoRebuildMode.MissingCheckpointsOnly;
```

### Task 3 — `ProjectionDaemon.cs`

Replace the startup block:
```csharp
if (_options.EnableAutoRebuild)
{
    await RebuildMissingProjectionsAsync(stoppingToken).ConfigureAwait(false);
}
```

With a switch covering all three modes:
```csharp
switch (_options.AutoRebuild)
{
    case AutoRebuildMode.MissingCheckpointsOnly:
        await RebuildMissingProjectionsAsync(stoppingToken).ConfigureAwait(false);
        break;
    case AutoRebuildMode.ForceFullRebuild:
        await RebuildAllProjectionsAsync(stoppingToken).ConfigureAwait(false);
        break;
    case AutoRebuildMode.None:
    default:
        break; // daemon starts but does not trigger any rebuild
}
```

`RebuildAllProjectionsAsync` is a new private method that calls
`_projectionManager.RebuildAllAsync(forceRebuild: true, cancellationToken)`.

### Task 4 — `ProjectionOptionsValidator.cs`

Remove the `EnableAutoRebuild` validation entirely — the validator currently just
confirms the bool accepts both `true` and `false`, which is a no-op test. All enum
values are valid; no range check is needed.

### Task 5 — `ProjectionOptionsValidationTests.cs`

Replace `Validate_EnableAutoRebuild_AcceptsBothValues(bool)` theory with a single test
covering all three `AutoRebuildMode` values:

```csharp
[Theory]
[InlineData(AutoRebuildMode.None)]
[InlineData(AutoRebuildMode.MissingCheckpointsOnly)]
[InlineData(AutoRebuildMode.ForceFullRebuild)]
public void Validate_AutoRebuild_AcceptsAllModes(AutoRebuildMode mode)
{
    var options = new ProjectionOptions { AutoRebuild = mode };
    var result = new ProjectionOptionsValidator().Validate(null, options);
    Assert.True(result.Succeeded);
}
```

### Task 11 — CHANGELOG.md

```markdown
### Changed

- **`ProjectionOptions.EnableAutoRebuild` (bool) replaced by `AutoRebuild` (AutoRebuildMode enum)** — breaking change.
  The boolean only supported two states; the new enum adds a third mode:
  - `None` — daemon starts without triggering any rebuild (was `false`)
  - `MissingCheckpointsOnly` — only projections with absent checkpoint files are rebuilt
    on startup (was `true`; this remains the default)
  - `ForceFullRebuild` — all projections are rebuilt from scratch on every startup
    (was not expressible with a boolean; useful for development/migration scenarios)

  **Migration:** replace `"EnableAutoRebuild": true/false` in `appsettings.json` with
  `"AutoRebuild": "MissingCheckpointsOnly"` / `"AutoRebuild": "None"`.
```
