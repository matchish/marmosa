# Corrupt-Ledger Silent Reset — Implementation Tasks

> **Target version:** 0.5.0
> **Roadmap item:** [#6 — Corrupt-Ledger Silent Reset](../future-plans/0.5.0-roadmap.md#6-corrupt-ledger-silent-reset)

---

## Summary

`LedgerManager.GetLastSequencePositionAsync` catches `JsonException` and returns `0`.
This causes the next `AppendAsync` to allocate positions starting at 1, silently
overwriting every committed event file from the beginning of the store.

The fix replaces the silent `return 0` with an explicit `InvalidOperationException` that
includes the ledger file path and actionable recovery guidance. This transforms a silent
catastrophic data loss into a loud, actionable failure.

---

## Affected code

There are **two** `catch (JsonException)` blocks in `GetLastSequencePositionAsync`:

1. **Line ~70** — inside the retry loop (attempts 0 to maxRetries-1)
2. **Line ~101** — in the final attempt outside the retry loop

Both must be changed.

## Task Detail

### Tasks 1–2 — Replace Silent Return with Explicit Throw

**Current code** (two locations in `LedgerManager.cs`):

```csharp
catch (JsonException)
{
    // Ledger is corrupt - return 0 and let it be rebuilt
    return 0;
}
```

**Replacement** (both locations):

```csharp
catch (JsonException ex)
{
    var ledgerPath = GetLedgerPath(contextPath);
    throw new InvalidOperationException(
        $"Ledger file at '{ledgerPath}' is corrupt and cannot be parsed. " +
        "This would cause silent data loss by overwriting events from position 1. " +
        "Delete the ledger file and restart to re-derive the position from the events directory, " +
        "or restore the ledger from backup.",
        ex);
}
```

**Note:** The `GetLedgerPath` helper already exists — it is called earlier in the method
to resolve the ledger path. The variable `ledgerPath` is local to the retry block so it
must be computed again in the catch, or hoisted to method scope.

### Tasks 3–5 — Unit Tests

Create a test that writes invalid JSON to the ledger file and calls
`GetLastSequencePositionAsync`. Assert:

- `InvalidOperationException` is thrown (not `JsonException`)
- The message contains the ledger file path
- `InnerException` is the original `JsonException`

### Task 6 — Regression Check

Run the existing `LedgerManagerTests` to confirm:
- A valid ledger file returns the correct position (unchanged)
- A missing ledger file returns 0 (unchanged — this is the `FileNotFoundException`
  path, not the `JsonException` path)
