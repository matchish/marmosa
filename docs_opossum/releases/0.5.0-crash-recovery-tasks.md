# Crash-Recovery Position Collision — Implementation Tasks

> **Target version:** 0.5.0
> **Roadmap item:** [#5 — Crash-Recovery Position Collision](../future-plans/0.5.0-roadmap.md#5-crash-recovery-position-collision)
> **Limitation doc:** [docs/limitations/crash-recovery-position-collision.md](../limitations/crash-recovery-position-collision.md)

---

## Summary

`AppendAsync` writes event files (step 7) before updating the ledger (step 9). A process
crash between steps 7 and 9 leaves orphaned event files at positions the ledger does not
know about. The next startup reads the stale ledger, allocates the same positions, and
`WriteEventAsync` silently overwrites the orphaned files — even with
`WriteProtectEventFiles = true` — because the write path strips `ReadOnly` before every
overwrite.

The fix makes `WriteEventAsync` **idempotent**: if the destination file already exists,
skip the write. After all event files are confirmed on disk, reconcile the ledger to the
actual highest on-disk position.

---

## Approach: Existence Check Before Overwrite (Option B from roadmap)

### Design

1. **`WriteEventAsync`** — before `File.Move(tempPath, filePath, overwrite: true)`, check
   `File.Exists(filePath)`. If the file exists, delete the temp file and return without
   overwriting. This makes every event write idempotent on restart.

2. **Ledger reconciliation on startup** — add a new method
   `ReconcileLedgerAsync(string contextPath)` to `LedgerManager` (or a new
   `StartupRecoveryManager` class). On startup, scan the events directory for the
   highest-numbered `*.json` file. If this is higher than the ledger's
   `LastSequencePosition`, update the ledger to match. This closes the gap between
   orphaned event files and the stale ledger.

3. **ReadOnly attribute stripping** — the existing code that strips `ReadOnly` before
   overwrite is only needed for the `AddTagsAsync` maintenance path (which legitimately
   re-writes existing events to add tags). The idempotent skip in step 1 ensures
   crash-orphaned files are never touched. The `AddTagsAsync` path should be the only
   caller that passes `overwrite: true` semantics; the normal append path should never
   overwrite.

### Why not Option A (ledger-first / WAL)

Option A moves the ledger update before step 7 (write-ahead log semantics). This is more
principled but significantly more complex — it requires a startup recovery scan that
detects positions in the ledger without corresponding event files and replays the writes.
It also changes the failure semantics of a crash between ledger update and event write.
Option B is simpler, sufficient, and introduces no new failure modes.

## Task Detail

### Task 1 — Idempotent Existence Check in `WriteEventAsync`

**Current code** (lines 65–77 of `EventFileManager.cs`):

```csharp
// If write protection is enabled and the destination already exists (maintenance
// rewrite via AddTagsAsync), remove the read-only attribute before the overwrite.
if (_writeProtect && File.Exists(filePath))
{
    var existing = File.GetAttributes(filePath);
    if ((existing & FileAttributes.ReadOnly) != 0)
        File.SetAttributes(filePath, existing & ~FileAttributes.ReadOnly);
}

// Atomic move (now safe - data is on disk if flushing is enabled)
File.Move(tempPath, filePath, overwrite: true);
```

**Required change:**

Before the `ReadOnly`-stripping block, add an idempotent check: if the destination file
already exists and this is a **normal append** (not a maintenance rewrite), the event was
already persisted by a previous run that crashed before updating the ledger. Delete the
temp file and return without overwriting.

This requires distinguishing between "normal append" and "maintenance rewrite". Options:
- Add an `overwrite` boolean parameter to `WriteEventAsync` (default `false`).
- Or check: if the file exists and `_writeProtect` is false, it's a maintenance rewrite;
  if `_writeProtect` is true and the file exists, the caller must explicitly opt in to
  overwrite.

The simplest approach: add an optional `bool allowOverwrite = false` parameter. The
normal append path passes `false` (default). `AddTagsAsync` passes `true`.

### Task 2 — Separate ReadOnly Stripping

The `ReadOnly`-stripping logic should only execute when `allowOverwrite` is `true`.
When `allowOverwrite` is `false` and the file exists, the method should skip the write
entirely. This is a refactoring of the existing code within `WriteEventAsync`, not a
new method.

### Task 3 — Ledger Reconciliation

Add to `LedgerManager`:

```csharp
/// <summary>
/// Scans the events directory for the highest-numbered event file and updates
/// the ledger if it is behind. This reconciles the ledger after a crash that
/// wrote event files but did not update the ledger.
/// </summary>
public async Task ReconcileLedgerAsync(string contextPath, string eventsPath)
```

Implementation:
1. `Directory.GetFiles(eventsPath, "*.json")` — get all event files.
2. Parse filenames to extract positions (`long.Parse(Path.GetFileNameWithoutExtension(f))`).
3. Find the maximum position.
4. Read current ledger position via `GetLastSequencePositionAsync`.
5. If max on-disk position > ledger position, call `UpdateSequencePositionAsync` with
   the on-disk maximum.
6. Log a warning when reconciliation occurs (an event was recovered from a crash).

### Task 4 — Call Reconciliation in AppendAsync

In `FileSystemEventStore.AppendAsync`, after acquiring the cross-process lock (line 98)
and before `GetNextSequencePositionAsync` (line 109), call:

```csharp
await _ledgerManager.ReconcileLedgerAsync(contextPath, GetEventsPath(contextPath))
    .ConfigureAwait(false);
```

This runs inside the lock so no other process can append concurrently. It is safe to
read the events directory and update the ledger atomically.

**Performance consideration:** The directory scan runs on every `AppendAsync` call. For
stores with many event files this could be slow. Mitigation: cache the reconciliation
result in memory (a boolean `_reconciled`) so it only runs once per process lifetime.
The reconciliation is only needed on the first append after startup.

End-to-end tests that use the full `IEventStore` API:
